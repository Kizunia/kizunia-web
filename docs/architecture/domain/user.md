# Purpose

The User model represents every authenticated builder on Kizunia.

Every person who signs into the platform is represented by exactly one User.

The User model serves two primary responsibilities:

- Authentication identity (through Better Auth)
- Public builder profile

It intentionally does **not** store application-specific content such as projects, teams, blogs, or notification preferences directly. Those concerns are modeled through relationships with dedicated entities.

---

# Design Philosophy

The User model should remain lightweight.

It represents **who the builder is**, not **everything the builder owns**.

The model should contain only stable identity and profile information.

Everything else should be represented using relationships.

Examples:

- A user owns projects.
- A user belongs to teams.
- A user authors blogs.
- A user receives verification badges.

These should not become embedded fields within the User model.

This approach keeps the model easy to understand, maintain, and extend.

---

# Authentication

Authentication is handled entirely by Better Auth.

Kizunia intentionally uses Better Auth's User model as the primary application user rather than creating a separate profile model.

This provides several advantages:

- Better integration with Better Auth.
- Simpler queries.
- Less synchronization.
- Fewer joins.
- Cleaner implementation.

Authentication-specific concerns such as sessions, OAuth accounts, email verification, and providers remain the responsibility of Better Auth.

---

# Responsibilities

The User model is responsible for:

- Authentication identity
- Public profile
- Public username
- Builder information
- Visibility settings
- Relationships to other entities

The User model is **not** responsible for:

- Portfolio generation
- Project information
- Team information
- Notification preferences
- Verification
- Statistics
- Social metrics

# Fields

## Identity

Identity fields uniquely identify a builder.

| Field | Required | Unique | Description |
|--------|----------|--------|-------------|
| id | Yes | Yes | Primary identifier generated by Better Auth. |
| email | Yes | Yes | Authentication email managed by Better Auth. |
| emailVerified | Yes | No | Email verification status managed by Better Auth. |
| name | Yes | No | Public display name. Uses Better Auth's `name` field. |
| username | Yes | Yes | Public username used throughout the platform and in profile URLs. |

---

## Username

The username is the permanent public identifier of a builder.
we will use the better auth username module.

Profile URLs follow the format:

```
/u/{username}
```

Examples:

```
/u/ishankulkarni13

/u/john

/u/sarah-dev
```

### Rules

- Unique
- Editable
- Lowercase
- Characters: `a-z`, `0-9`, `-`
- No spaces
- Reserved usernames are prohibited
- Reasonable length restrictions apply

The username replaces the need for a separate user slug.

---

## Profile

These fields describe the builder.

| Field | Required | Description |
|--------|----------|-------------|
| image | No | Profile picture. Initially provided by Better Auth but may later be updated by the user. |
| banner | No | Banner image displayed on the public profile. |
| headline | No | Short professional introduction displayed below the user's name. |
| bio | No | Longer biography describing the builder. |
| college | No | Educational institution. |
| degree | No | Degree or academic program. |
| graduationYear | No | Expected graduation year. |
| location | No | Optional public location. |

## System

System fields are managed by the platform.

| Field | Required | Description |
|--------|----------|-------------|

# Visibility

Every profile supports one visibility level.

Possible values:

- Public
- Unlisted
- Private

Visibility controls how easily a builder can be discovered on Kizunia.

---

# Status

The current version supports two statuses.

- Active
- Suspended

Additional statuses should only be introduced when a clear product requirement exists.

# Relationships

The User model intentionally owns very little information.

Most application functionality is expressed through relationships.

## Projects

A user may:

- Own projects
- Maintain projects
- Contribute to projects

Each relationship is independent.

Being a contributor does not imply ownership.

Being an owner does not imply authorship.

---

## Teams

A user may:

- Own teams
- Maintain teams
- Join teams

Membership and ownership remain separate concepts.

---

## Hackathons

A user may:

- Bookmark hackathons
- Participate in hackathons through teams and projects

Hackathon participation is modeled through relationships rather than direct ownership.

---

## Blogs

A user may author multiple blogs.

Blog moderation and publishing are handled independently from the User model.

---

## Links

External links are modeled through the reusable Link entity.

Examples include:

- GitHub
- LinkedIn
- Portfolio
- Devpost
- YouTube

The User model stores relationships to links rather than dedicated URL fields.

---

## Technologies

Skills are represented through relationships with the Technology entity.

Examples include:

- React
- Flutter
- Spring Boot
- TensorFlow

This enables consistent filtering, recommendations, and search throughout the platform.

---

## Interests

Builder interests are represented through Categories.

These interests power personalized hackathon recommendations and notifications.

---

## Notification Preferences

Notification preferences are stored separately.

Examples include:

- Preferred technologies
- Preferred categories
- Preferred locations
- Online / Offline preference
- Notification channels

Separating these preferences keeps the User model focused on identity rather than configuration.

---

## Verification

Verification is modeled using a dedicated Verification entity.

A user may possess multiple verification badges.

Examples include:

- Kizunia Owner
- Kizunia Administrator
- Verified Organizer
- Verified Project Owner

The User model never stores simple boolean verification flags.

# Design Decisions

## Why Better Auth's User Model?

Using Better Auth's User model directly provides:

- Better framework integration.
- Simpler architecture.
- Reduced synchronization.
- Fewer joins.
- Easier maintenance.

Creating a separate UserProfile model would introduce unnecessary complexity without solving a real problem for the current scope of Kizunia.

---

## Why No User Slug?

Usernames already provide a unique, human-readable identifier.

Adding a separate slug would duplicate responsibility and introduce unnecessary complexity.

The public URL always uses:

```
/u/{username}
```

---

## Why No Statistics?

Statistics such as:

- Project count
- Team count
- Contribution count

are derived from relationships.

They should never be stored directly on the User model.

---

## Why No Social Features?

Kizunia is a builder-first platform.

The platform intentionally avoids:

- Followers
- Following
- Likes
- Reputation
- Profile views

These metrics do not contribute to the core mission of helping builders discover opportunities, collaborate, and preserve their work.

---

# Future Expansion

The User model has been intentionally designed to allow future growth.

Possible future additions include:

- Resume generation
- Public achievements
- Portfolio themes
- Custom portfolio domains
- Developer activity timeline

These additions should build upon existing relationships rather than expanding the User model unnecessarily.

---

# Guiding Principle

The User model should answer one question:

> **Who is this builder?**

It should never attempt to answer:

> **What has this builder built?**

Those answers belong to Projects, Teams, Hackathons, Blogs, and other related entities.

Keeping this separation ensures the User model remains simple, maintainable, and aligned with the long-term architecture of Kizunia.
